# Deadliner UI 专项治理技术方案

## 1. 背景

本轮 UI 专项治理已经发生了一次明确转向：

- 不再以“Material 3 与 MIUIX 双栈并行收口”作为最终目标。
- Material 3 作为正式设计系统继续推进。
- MIUIX 相关尝试视为一轮探索性分支，本期不再作为正式版交付目标。

因此，这份文档也同步从“融合 Material 3 + MIUIX”改为“统一外观状态、收口 Material 3、补齐显示大小与图标能力”的真实执行方案。

## 2. 当前目标

当前版本的治理目标收敛为三件事：

1. 统一并优化 Material 3 主题与色彩架构，消除历史上由 MIUIX 混入带来的状态分裂和颜色异常。
2. 完成“应用显示大小”能力，提供稳定的预设缩放，并保留实验性的自定义比例入口。
3. 上线 iOS 已有的“季节图标”和“自定义图标”能力，在 Android 上提供稳定一致的桌面图标体验。

## 3. 已完成进展

### 3.1 外观状态建模已收口

已经完成的内容：

- 新增统一外观模型 `AppearancePreferences`。
- 将 `style`、`seedColor`、`displayScalePreset`、`appIconMode` 等外观能力纳入同一配置结构。
- `GlobalUtils` 已开始作为兼容层，逐步代理到新的外观状态。
- `MainActivity`、设置页、引导页不再依赖“只读一次”的零散外观字段。

当前结论：

- “页面风格”和“主题状态”已经不再继续向 MIUIX 方向扩展。
- 后续治理应继续减少旧字段和兼容入口，而不是再加新的布尔分支。

### 3.2 色彩内核 Phase 1 已基本落地

已经完成的内容：

- 已建立统一 token 路径：
  - `DeadlinerColorTokens`
  - `DeadlinerColorTokenFactory`
  - `DeadlinerMaterial3ColorSchemeFactory`
- `Theme.kt` 已改为从统一 token 派生 Material 3 色板。
- `miuixColor` 这类历史配置已退出有效产品能力，不再作为正式设置项暴露。

当前结论：

- 主题治理的主线已经变成“单一 Material 3 语义色内核”。
- 文档和实现都不应再把 MIUIX 当作正式目标色彩体系。

### 3.3 显示大小能力已经具备基础上线形态

已经完成的内容：

- 已落地 `DisplayScalePreset`、`DisplayScaleManager`、Base Activity 级显示密度覆写。
- 主要 Activity 已接入显示大小上下文。
- 设置结构已经调整为：
  - “界面外观”下提供二级菜单形式的显示大小预设。
  - “实验室”中保留“自定义显示大小缩放比例”入口。
- 支持预设档位与实验性自定义比例。

当前结论：

- 显示大小已经不是“从 0 到 1”的能力，而是进入“参数校准、交互打磨、回归验证”阶段。

### 3.4 ModernHost 顶栏治理已经开始

已经完成的内容：

- `ModernHost` 顶栏切换已具备过渡动画。
- `TextPageIndicator` 已加入选中态动画。
- 列表滚动时，顶部次级选择区支持自动收起。

当前结论：

- `ModernHost` 已进入具体页面打磨阶段。
- 后续重点是统一顶栏层级、AI 入口位置、次级操作条节奏，而不是继续引入新设计系统。

## 4. 不再继续推进的方向

以下内容不再作为本期目标：

1. 不再把 MIUIX 做成正式版界面。
2. 不再继续投入“Material 3 / MIUIX 双栈视觉对齐”的专项调试。
3. 不再对外暴露 `MIUIX Mode`、`MIUIX 颜色` 之类历史配置概念。

处理原则：

- 代码层允许保留短期兼容壳，但不再继续扩张。
- 文档层不再把 MIUIX 写成正式交付目标。
- 后续有机会时，应逐步删除旧兼容路径和残余分支。

## 5. 现阶段总体技术方案

### 5.1 外观状态统一

继续以 `AppearancePreferences` 作为唯一外观配置模型，统一管理：

- `uiStyle`
- `themeStyle`
- `colorSource`
- `displayScalePreset`
- `customDisplayScaleMultiplier`
- `appIconMode`

治理原则：

- 外观配置必须有单一状态源。
- 历史字段仅作为迁移兼容层，不再作为产品配置入口。
- 新功能必须先接入统一外观状态，再接 UI。

### 5.2 色彩治理统一到 Material 3

当前正式方案：

1. 先生成 `DeadlinerColorTokens`。
2. 由 token 派生 Material 3 色板。
3. 页面和基础组件只消费语义色，不直接依赖旧的历史开关。

治理原则：

- 不再为 MIUIX 单独维护正式色彩策略。
- `DynamicColorsExtension`、旧 overlay、旧颜色开关都应逐步降级为兼容层。
- `ui/base/*`、设置页、主界面、详情页都应统一走 token 语义色。

### 5.3 显示大小方案

正式方案保持不变：通过应用级 `Configuration.densityDpi` 覆写实现。

产品策略：

1. 正式设置中只提供预设档位。
2. 实验室只保留“自定义显示大小缩放比例”入口。
3. 不改系统字体缩放，不额外引入独立字体大小能力。

治理原则：

- Compose 与 View 页面必须一起生效。
- 多窗口、旋转、重启后状态必须稳定。
- Widget 不跟随应用内显示大小。

### 5.4 应用图标方案

正式方案保持不变：采用 `activity-alias + PackageManager#setComponentEnabledSetting(...)`。

本期产品定义：

- 默认图标
- 季节自动图标
- 手动季节图标
- 官方预置自定义图标

治理原则：

- 自定义图标仅指官方预置图标集合，不做用户上传任意图片。
- 任意时刻只允许一个 launcher alias 处于启用状态。
- 图标能力纳入统一外观配置，不单独散落在其他偏好字段里。

## 6. 剩余工作清单

### 6.1 Phase A：清理 MIUIX 遗留与文档口径

目标：

- 把 MIUIX 从“计划目标”彻底降级为“历史遗留兼容项”。

剩余工作：

- 清理文档、设置文案、注释中仍把 MIUIX 视为正式目标的表述。
- 梳理 `GlobalUtils`、`Theme.kt`、`ui/base/*` 中残余的 MIUIX 兼容逻辑。
- 标记哪些兼容代码本期保留，哪些可以直接删除。

### 6.2 Phase B：Material 3 色彩与组件收口

目标：

- 完成正式设计系统下的颜色和基础组件统一。

剩余工作：

- 继续清理直接读取旧颜色字段的组件分支。
- 收口 `TopAppBar`、`Button`、`Switch`、`Checkbox`、`Slider`、`Dialog` 等基础组件的颜色与尺寸策略。
- 对 `ModernHost`、`SimplifiedHost`、设置页、详情页做可视一致性检查。

### 6.3 Phase C：显示大小上线前打磨

目标：

- 把显示大小从“可用”推进到“稳定可上线”。

剩余工作：

- 校准预设档位倍率。
- 实机验证高 DPI 小屏、大屏、分屏、多窗口、旋转场景。
- 检查关键页面：
  - 主界面
  - 设置页
  - 添加页
  - 详情页
  - 概览页
  - Widget 配置页
- 决定自定义比例的范围、交互文案、重启策略是否还需要细化。

### 6.4 Phase D：应用图标能力上线

目标：

- 补齐 Android 图标切换基础设施并接到设置页。

剩余工作：

- 增加 launcher alias 结构。
- 新增 `AppIconManager`。
- 新增季节解析和自动切换逻辑。
- 接入设置页图标预览和模式选择。
- 补齐图标资源与回归验证。

### 6.5 Phase E：ModernHost 视觉打磨

目标：

- 把 `ModernHost` 顶栏和主界面节奏做得更完整。

剩余工作：

- 定稿 `问 AI` 的最终产品位置和形态。
- 继续打磨顶部主栏与次级选择区的视觉层级。
- 统一总览页、拍摄页、列表页的顶栏动作区语言。
- 检查滚动收起、切页过渡、搜索态之间的衔接是否足够自然。

## 7. 风险与注意事项

### 风险 1：显示大小与个别页面测量逻辑冲突

表现：

- 某些 Compose 页面或旧 XML 页面在高缩放下可能出现裁切、换行异常或点击区域偏差。

应对：

- 建立重点页面回归矩阵。
- 优先修正固定高度、固定宽度、写死间距的问题。

### 风险 2：历史兼容逻辑继续拖慢治理

表现：

- 旧字段还在写，新的统一状态也在写，导致行为再次分裂。

应对：

- 统一要求新增 UI 能力只能接 `AppearancePreferences`。
- 对旧字段加注释和迁移说明，逐步清理。

### 风险 3：图标切换在不同 Launcher 上刷新延迟不一致

表现：

- 不同 ROM、不同 Launcher 对 alias 切换刷新速度不同。

应对：

- 提前实机验证主流机型。
- 切换后给出轻提示，不承诺绝对实时刷新。

## 8. 验收标准

本轮专项治理完成时，至少满足以下条件：

1. 文档、设置、代码实现都不再把 MIUIX 作为正式版目标。
2. 外观配置入口统一，不再继续扩张历史布尔开关。
3. Material 3 色彩与基础组件在核心页面表现一致，无明显主题异常。
4. 应用显示大小支持稳定的预设档位，设置入口和实验室入口结构清晰。
5. 季节图标与官方预置自定义图标能力完成上线。
6. `ModernHost` 的顶栏、次级选择区、AI 入口达到可接受的正式版观感。

## 9. 推荐执行顺序

建议按以下顺序推进：

1. 先完成文档和代码口径统一，把 MIUIX 从正式目标中移除。
2. 再收 Material 3 基础组件与色彩一致性。
3. 然后完成显示大小的校准与回归。
4. 接着推进应用图标能力。
5. 最后集中做 `ModernHost` 和核心页面的视觉打磨。

这样做的好处是：

- 路线更清晰，不会继续在已经转向的 MIUIX 目标上消耗时间。
- 剩余工作会集中在真正能上线的能力上。
- 主界面打磨可以建立在已经稳定的外观和显示大小基础上，返工更少。
